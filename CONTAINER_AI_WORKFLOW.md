# 📖 AI 容器化與高階 Agent 數據情報工作流知識庫（v2 修訂版）

## 🔍 修訂說明

本版針對初稿進行了結構性校正，重點如下：

1. **修正沙盒比喻的邏輯方向**：沙盒的本質是「限制內部執行風險」，而非「阻擋外部入侵」，因此比喻從「外圍鐵絲網」改為「房子內部的抗爆隔離艙」。
2. **補上缺失的 Agent 角色**：原稿工作流第 4 步描述「正反方辯證分析」，但先前定義的 Subagent 只有工程師與測試員，缺乏對應角色 → 新增「決策分析小組」。
3. **建立概念與流程的一致映射**：將第一部分的四大基礎概念，與第三部分的五步驟工作流做逐一對應，使字典與流程互相印證，而非兩份各自獨立的文件。
4. **去除重複敘述**，將條列式說明改為表格化、流程化呈現。

## 🧱 第一部分：核心設計原則 — 四層基礎設施

整套系統可以濃縮為一句話：

> **先隔離，再賦能，接著安全連結外部世界，在受控環境中驗證邏輯，最終產出可信任的決策。**

這句話對應四層基礎設施，由內而外分別是：

```
隔離層 (Container)
   └─ 能力層 (Package / Plugin)
        └─ 通訊層 (Connector)
             └─ 安全層 (Sandbox)
```

| 層級 | 概念 | 建築比喻 | 核心功能 | 在 AI 專案中的角色 |
| :--- | :--- | :--- | :--- | :--- |
| 隔離層 | **容器 Container** | 一棟獨立的房子 | 將應用程式與作業系統底層、運行環境打包成一致的隔離空間 | 確保 AI 模型在任何電腦或雲端都能跑出相同結果 |
| 能力層 | **套件 Package** | 蓋房子的磚頭 | 預先寫好、不能獨立執行的功能函式庫 | 提供基礎能力，如 `pandas`（資料分析）、`playwright`（爬蟲） |
| 能力層 | **外掛 Plugin** | 房子的裝潢 | 依附主程式運作，用來擴充特定功能 | 補足主程式沒有的專項能力，如 PDF 解鎖、多模態視覺處理 |
| 通訊層 | **連接器 Connector** | 外接的水電管線 | 在不同系統／平台之間牽線、傳遞資料，本身不執行邏輯 | 讓 AI 安全串接外部世界，如資料庫、雲端硬碟、外部 API |
| 安全層 | **沙盒 Sandbox** | 房子內部的抗爆隔離艙 | 開闢一個極低權限的受限區域，專門承接未知或有風險的執行行為 | AI 生成的程式碼在此執行，出錯或失控都不會波及核心系統 |

> 「容器」界定邊界、「套件／外掛」提供能力、「連接器」負責對外連結、「沙盒」負責對內止損——四者分工明確，缺一不可。

## 🤖 第二部分：Agent 協作架構 — 雙小組制

高階 Agent（如 Claude Code）處理宏觀任務時，會依「目的」而非單一角色拆分成兩支獨立小組，各自形成閉環：

### 小組 A｜技術實作小組（Engineering Squad）
負責讓程式碼「能跑」，形成**自癒閉環**：

| Subagent | 職責 |
| :--- | :--- |
| 工程師 (Code Generator) | 理解需求與上下文，撰寫自動化腳本或分析程式碼 |
| 測試員 (Code Executor) | 在沙盒中執行程式碼、捕捉錯誤訊息，回饋給工程師修正 |

```
工程師寫 Code → 測試員丟進沙盒執行 → 報錯 → 工程師修正 → 重新執行（直到成功）
```

### 小組 B｜決策分析小組（Analyst Squad）
負責針對「數據代表的意義」進行**辯證推論**，而非寫程式：

| Subagent | 職責 |
| :--- | :--- |
| 正方分析師 (Proponent) | 從技術可行性與市場機會角度，論證數據的正面意義 |
| 反方分析師 (Critic) | 從風險、時程延遲、商業衝擊角度，提出質疑與反例 |
| 綜合仲裁者 (Synthesizer) | 彙整正反論點，收斂成可執行的結論與信心水準 |

> 兩支小組共用同一個沙盒環境執行與驗證，但目的不同：技術小組驗證「程式碼能不能跑」，分析小組驗證「結論站不站得住腳」。

## 🚀 第三部分：標準工作流 SOP（五步驟）

四層基礎設施 + 兩支 Agent 小組，共同組成一條完整的情報產出流水線：

```
[1] Container      [2] Package/Plugin   [3] Connector        [4] Sandbox              [5] 產出
建立隔離環境   ──▶  補齊工具與外掛   ──▶  安全抓取外部數據  ──▶  雙小組驗證與辯證   ──▶  BI 呈現與決策建議
```

| 步驟 | 對應概念 | 執行內容 | 涉及角色 | 產出物 |
| :-- | :--- | :--- | :--- | :--- |
| 1. 環境初始化 | Container | 以專案目錄為邊界，啟動獨立容器，隔離本機系統 | — | 乾淨的作戰環境 |
| 2. 工具與技能載入 | Package / Plugin | 自主安裝資料分析、爬蟲等套件，載入特定格式外掛 | — | 具備完整能力的執行環境 |
| 3. 安全資料牽線 | Connector | 定時抓取主題新聞，與 `knowledge_base/` 歷史資料比對差異 | — | 每日差異摘要（晨報素材） |
| 4. 沙盒驗證與辯證 | Sandbox | 技術小組跑通模型程式碼；分析小組在沙盒模擬結果上進行正反辯證 | 技術實作小組 + 決策分析小組 | 通過驗證的模型結果、正反論點 |
| 5. 終端輸出 | （產出目標，非基礎設施概念） | 將驗證結果渲染成網頁 BI 圖表，彙整辯證結論 | 綜合仲裁者 | 趨勢圖表 +「建議決策行動」與「應持續追蹤議題」 |

> 注意第 5 步並不對應任何一個基礎設施概念，它是前四層共同服務的**最終目的**——沒有產出，前面的隔離、賦能、連結、驗證都沒有意義。

## 💡 第四部分：實戰指令範本

使用 Claude Code 時，直接以下列「架構語意」下達指令，而非描述實作細節：

> **🎯 企業級自動化情報系統建立指令：**
>
> 「我想針對 **[主題，例如：固態電池技術商用化]** 建立一個每日晨報與決策系統，請在目前的專案容器內建立全自動工作流：
>
> 1. **（Connector）** 配置連接器每日抓取新聞，與 `knowledge_base/` 資料夾的歷史資料比對差異，產出晨報摘要。
> 2. **（Analyst Squad）** 啟動決策分析小組，由正方分析師與反方分析師針對新舊數據進行辯證（技術可行性、市場衝擊、時程），最後由綜合仲裁者收斂結論。
> 3. **（Sandbox + Engineering Squad）** 所有驗證與運算腳本，必須在沙盒環境中執行，若出錯由工程師與測試員自我修正，直到成功運行。
> 4. **（產出）** 成功後於專案內產出網頁 BI 圖表呈現趨勢，並在終端機輸出『建議決策行動』與『應持續追蹤議題』總結報告。」

### 精簡版（快速觸發用）

> 「針對 **[主題]**，在容器內建立每日晨報系統：連接器抓新聞比對歷史庫 → 分析小組正反辯證 → 沙盒驗證程式碼 → 輸出 BI 圖表與決策建議。」
